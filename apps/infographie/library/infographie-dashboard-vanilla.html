🏗️ Architecture Dashboard en Vanilla JavaScript

Structure complète et bonnes pratiques pour un dashboard sans framework

🎯 Vanilla JS = Simple, Rapide, Contrôlable

Un dashboard en HTML/CSS/JS pur est parfaitement viable pour des projets petits à moyens (1-5 vues, <30 composants).
L'important : adopter une architecture claire dès le départ.

🤔 Quand choisir Vanilla JavaScript ?

✅ Vanilla est un bon choix si :

  • 1-5 vues principales, moins de 30 composants UI
  • Données modestes (quelques milliers de lignes max)
  • Petite équipe (1-3 devs) ou projet solo
  • Besoin de chargement ultra-léger
  • Logique d'état simple, peu de workflows complexes
  • Mise en production rapide

⚠️ Considérer un framework si :

  • Nombreux écrans et état imbriqué complexe
  • Très grands tableaux (>10k lignes, virtualisation)
  • Temps réel intensif avec multiples sources
  • Équipe >3 devs (conventions et tooling)
  • Internationalisation et theming avancé
  • Accessibilité à grande échelle

🧱 Les 9 Piliers de l'Architecture Vanilla

📦
1. Modules ES
Séparez votre code en fichiers distincts selon les fonctionnalités. Plus de fichier monolithique !
Méthode : import/export
Organisation : Un fichier par page/feature
Exemple : auth.js, dashboard.js, api.js
🧩
2. Composants Réutilisables
Créez des blocs UI réutilisables pour éviter la duplication de code.
Options :
• Web Components (Custom Elements)
• Fonctions render() simples
• Templates <template> natifs
Exemples : UserCard, Button, Alert
🎨
3. Variables CSS & Thèmes
Centralisez vos couleurs et styles dans des variables CSS pour faciliter la maintenance.
Méthode : CSS Custom Properties
Avantage : Dark mode facile
Exemple : --primary-color, --bg-color
💾
4. Gestion d'État Légère
Un petit système pour mémoriser les données importantes (utilisateur, filtres, etc.).
Approches :
• Objet JS simple avec EventTarget
• CustomEvent pour broadcast
• Pas besoin de Redux !
🧭
5. Navigation (Routing)
Changez de page sans recharger tout le site pour une expérience fluide.
Options :
• Hash routing : #/dashboard
• History API : pushState()
Résultat : SPA (Single Page App)
🌐
6. Récupération de Données
Utilisez fetch() natif pour communiquer avec votre backend.
Méthode : fetch() + async/await
Cache : localStorage / indexedDB
Contrôle : AbortController
♿
7. Accessibilité (A11y)
Rendez votre dashboard utilisable par tous, y compris les personnes en situation de handicap.
Bonnes pratiques :
• Balises sémantiques HTML
• Rôles ARIA appropriés
• Navigation au clavier
• Contrastes suffisants
🧪
8. Tests & Qualité
Testez votre code pour éviter les bugs et maintenir la qualité.
Outils simples :
• Tests unitaires (Vitest)
• Lighthouse (Chrome DevTools)
• axe pour l'accessibilité
⚡
9. Optimisation (facultatif)
Optimisez votre code pour un chargement plus rapide.
Outils : Vite, Parcel
Actions :
• Minification
• Transpilation
• Code splitting

🎨 Variables CSS

💾 State Management Simple

class Store extends EventTarget { constructor() { super() this.state = {} } setState(key, value) { this.state[key] = value this.dispatchEvent( new CustomEvent('change', { detail: { key, value } }) ) } getState(key) { return this.state[key] } } const store = new Store() export default store

🧭 Router Hash Simple

const routes = { '/': HomePage, '/dashboard': DashboardPage, '/analytics': AnalyticsPage } function router() { const path = window.location.hash .slice(1) || '/' const page = routes[path] if (page) page.render() } window.addEventListener( 'hashchange', router ) router() // Init

🌐 Fetch API avec Cache

async function fetchData(url) { // Check cache const cached = localStorage .getItem(url) if (cached) { return JSON.parse(cached) } // Fetch const res = await fetch(url) const data = await res.json() // Cache localStorage.setItem( url, JSON.stringify(data) ) return data }

📊 Tableaux & Graphiques : Choisissez des Micro-Libs

Pour les tableaux complexes et graphiques, utilisez des bibliothèques spécialisées légères comme Chart.js, D3.js, ou Plotly pour les visualisations, et des libs de tableaux avec tri/filtre pour les données tabulaires. Ces outils ne compromettent pas votre approche vanilla !

✅ Checklist de Réussite

  • Responsive Design : Grid/Flexbox pour tous les écrans
  • Thème Clair/Sombre : Variables CSS + media query
  • Accessibilité : Sémantique HTML + ARIA + navigation clavier
  • Performance : Lazy-load des vues, éviter DOM thrashing
  • State Management : Store simple avec EventTarget
  • Routing : Hash ou History API pour navigation fluide
  • API Layer : fetch() centralisé avec gestion d'erreurs
  • Cache Local : localStorage ou indexedDB pour données fréquentes
  • Feedbacks UI : Skeletons, toasts, validations en ligne
  • Tests : Unitaires + Lighthouse pour la qualité
  • Sécurité : Validation client ET serveur, protection XSS
  • Build Optionnel : Vite/Parcel pour optimisation

🧰 Micro-Libs Vanilla-Friendly (optionnelles)

📅 Day.js

Manipulation de dates légère (alternative à Moment.js). Parfait pour formater des timestamps.

📊 Chart.js

Graphiques simples et élégants. Idéal pour les dashboards avec visualisations basiques.

🎨 D3.js

Visualisations de données avancées et personnalisables. Pour des besoins plus complexes.

🔄 SortableJS

Drag & drop natif pour listes et tableaux réorganisables.

💾 FileSaver.js

Exports de fichiers côté client (CSV, PDF, etc.).

⚡ Alpine.js

Réactivité légère dans le HTML. Alternative minimaliste à Vue pour petites interactions.

🎯 En Résumé : Architecture Vanilla Réussie

1. Modules ES pour l'organisation → 2. Composants réutilisables → 3. Variables CSS pour le thème → 4. State management léger → 5. Routing pour la navigation → 6. fetch() pour les données → 7. Accessibilité dès le départ → 8. Tests avec outils simples → 9. Build optionnel pour optimiser

🔑 Le principe clé : Structure simple + conventions claires = Code maintenable sans framework !
